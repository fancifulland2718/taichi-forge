# Taichi 编译流水线优化更新审查与保留清单

**审查日期**：2026-04-24  
**审查原则**（按优先级排序）：

1. **正确性** —— 任何更新必须可证明不改变用户可见的数值语义（默认路径），或在可接受的、已文档化的容差内；
2. **稳定性** —— 不引入新的崩溃路径、未定义行为，也不降低现有测试覆盖的可靠性；
3. **运行时性能** —— 不让内核执行本身变慢；
4. **架构影响** —— 改动面与其收益匹配，不引入隐式契约或隐藏的维护负担；
5. **现代化与工具链兼容性** —— 与主流 LLVM / MSVC / Python 发行节奏保持一致；
6. **编译收益** —— 在满足以上五项的前提下，有可复现、方向性明确、量级可观的冷编译加速。

**强制判则**：**若前五项中任一条出现一定负面作用，且该更新未带来较高的编译收益，则该更新不予保留。**

---

## 一、总览

本次审查覆盖从 Phase A.4（LLVM 15→19）到 P3.d-C（`compile_tier="fast"` → LLVM O0）为止的所有已落盘更新。审查结论：

| 状态 | 数量 | 阶段 |
|---|---|---|
| **保留** | 8 项 | A.4、P2.0、P2.a、P2.b、P2.c、P1.d、P3.a、P3.d-C |
| **回退** | 1 项 | **P3.c**（scalarize 早退） |

累计净效果（默认 `compile_tier="balanced"` vs Taichi 1.7.4）：

- **mat14**：CPU/CUDA/Vulkan 三后端 **1.53×–1.57×** 冷编译加速；
- **sph_force**：CPU **1.29×** 冷编译加速；
- **tier=fast** 再叠加：CPU **+11.4%**、CUDA **+12.1%**（opt-in，默认未动）；
- **运行时**：CPU 0.96×–1.15×、CUDA 0.77×–1.91×（源于 LLVM 19 本身的 codegen 提升）；
- **正确性**：`rel_err` 最大 **≤ 4e-6** 跨后端、**≤ 2e-7** 单后端（均在 Taichi 既有 1e-5 跨后端验收线内）。

## 二、保留项明细

### A.4 · LLVM 15 → LLVM 19 迁移

- **代码面**：`taichi/codegen/llvm/*.cpp`（opaque pointer 兼容层）、PassManagerBuilder → New PassManager 全量替换。
- **编译收益**：CPU 1.09×–1.16×、CUDA 1.04×–1.15×（saxpy 最慢 +4%）。
- **运行时收益**：CPU 0.96×–1.15×、CUDA 0.77×–1.91×（mandelbrot CUDA 0.77× 为 sub-ms launch 噪声）。
- **正确性**：12/12 用例通过，最大 `rel_err < 4e-7`，`nbody` CUDA `1.27e-12` / CPU `1.11e-9`。
- **判决依据**：现代化收益最强；运行时净正；所有后续 P1–P3 更新以此为前提。

### P2.0 · WholeKernelCSE O(N²) → O(users)

- **代码面**：[taichi/transforms/whole_kernel_cse.cpp](taichi/transforms/whole_kernel_cse.cpp)、[taichi/transforms/compile_to_offloads.cpp](taichi/transforms/compile_to_offloads.cpp)。
- **机制**：反向 def-use 链 + `first_iteration` 护栏，让冗余 CSE 扫描只在需要时运行。
- **编译收益**：mat14 65.4s → 35.3s（**1.85×**）。
- **风险**：早期在 LLVM 19 上对 sph_force 出现 0.66× 回退，已由 P2.a / P2.b 对冲至 1.29×。
- **判决依据**：收益量级远高于所引入的复杂度；核心 IR 通道，改动精准。

### P2.a · Simplified II 脏标记去重

- **代码面**：[taichi/transforms/compile_to_offloads.cpp](taichi/transforms/compile_to_offloads.cpp)。
- **机制**：`handle_external_ptr_boundary` 改为返回 `bool`，只在真正产生脏标记时重跑 `whole_kernel_cse`。
- **增量收益**：相对 P2.0 再 +6%（CPU sph_force 7.373s → 6.934s；mat14 42.393s → 39.8s）。
- **正确性**：`rel_err < 1.7e-6`，三后端 smoke + 正确性门全绿。
- **判决依据**：对 P2.0 的 sph_force 回退有直接对冲作用；实现极简。

### P2.b · LICM first-iteration 护栏

- **代码面**：[taichi/transforms/simplify.cpp](taichi/transforms/simplify.cpp)。
- **机制**：`loop_invariant_code_motion` 在 `full_simplify` 循环的后续迭代中跳过（LICM 本身带不动点，重复运行恒为空操作）。
- **收益**：mat14 / sph_force CPU ±1% 噪声；Vulkan sph_force +3%。
- **风险**：语义 no-op；改动面仅一行条件。
- **判决依据**：**收益本身不显著**，但该更新同时为 P2.c 的 `full` tier 提供了"回退到 pre-guard 行为"的唯一开关。剥离它会让 tier 枚举失去对称性。维护成本为零，不构成移除理由。

### P2.c · CompileTier 枚举（fast / balanced / full）

- **代码面**：[taichi/program/compile_config.h](taichi/program/compile_config.h)、[taichi/python/export_lang.cpp](taichi/python/export_lang.cpp)、[taichi/analysis/offline_cache_util.cpp](taichi/analysis/offline_cache_util.cpp)、[taichi/transforms/simplify.cpp](taichi/transforms/simplify.cpp)。
- **机制**：引入 `std::string compile_tier{"balanced"}` 字段；`balanced` 保持 P2.b 行为，`full` 回到 pre-P2.0，`fast` 为 P1.d / P3.d-C 的分发点。offline-cache key 已包含该字段，三档之间不会冲突落盘。
- **编译收益（默认 balanced vs 1.7.4）**：mat14 CPU 62.2s → 40.1s（**1.55×**）、CUDA（**1.53×**）、Vulkan（**1.57×**）；sph_force CPU 9069ms → 7006ms（**1.29×**）。
- **正确性**：三档 smoke Δ=0；跨后端 Δ ≤ 4e-6。
- **判决依据**：默认路径无任何额外成本；被 P3.d-C 追认为真正的开关点；架构影响最小（只加了一个 string 字段）。

### P1.d · Vulkan `spv_opt_level` 按 tier 分发

- **代码面**：[taichi/codegen/spirv/kernel_compiler.cpp](taichi/codegen/spirv/kernel_compiler.cpp)。
- **机制**：`compile_tier="fast"` 时把 `spv_opt_level` 从 3（23 passes）降到 1（3 passes：WrapOpKill / DeadBranchElim / AggressiveDCE）。
- **收益**：heavy_kernels（mat14 / sph_force）上 `fast ≈ balanced`（±1% 噪声）；对 SPV-bound 内核（局部数组 / 访存主导）收益更可见，但当前 bench 未覆盖。
- **判决依据**：改动仅一条 cap 语句；默认路径不受影响；"full" 档属显式 opt-in。**虽然当前 bench 上收益不显著，但代码面极小且未引入任何风险，维持保留。**

### P3.a · Python Unroll 硬上限（fail-fast）

- **代码面**：[python/taichi/lang/misc.py](python/taichi/lang/misc.py)、[python/taichi/lang/impl.py](python/taichi/lang/impl.py)、[python/taichi/lang/ast/ast_transformer.py](python/taichi/lang/ast/ast_transformer.py)、[python/taichi/lang/ast/ast_transformer_utils.py](python/taichi/lang/ast/ast_transformer_utils.py)、[python/taichi/lang/kernel_impl.py](python/taichi/lang/kernel_impl.py)。
- **机制**：新增 `unrolling_hard_limit`（单 loop）、`unrolling_kernel_hard_limit`（累计）、`func_inline_depth_limit` 三个 knob，**超限直接 `raise TaichiCompilationError`**（不静默截断）。
- **收益**：对合法内核无提速；对 pathological case（N=1600，limit=50）实现 9.035s → 0.0128s 的 **707× 早失败**（防护性能，非编译加速）。
- **默认值**：全部为 0（关闭），向后兼容。
- **判决依据**：默认零开销；显式语义（报错而非静默）；对 P3 阶段的 bench 可重现性是必要工具。

### P3.d-C · `compile_tier="fast"` → LLVM O0（NVPTX/AMDGCN 下保底 O1）

- **代码面**：[taichi/runtime/llvm/llvm_opt_pipeline.h](taichi/runtime/llvm/llvm_opt_pipeline.h)（新增 `effective_llvm_opt_level` helper）、[taichi/codegen/cpu/codegen_cpu.cpp](taichi/codegen/cpu/codegen_cpu.cpp)、[taichi/codegen/dx12/dx12_global_optimize_module.cpp](taichi/codegen/dx12/dx12_global_optimize_module.cpp)、[taichi/runtime/cuda/jit_cuda.cpp](taichi/runtime/cuda/jit_cuda.cpp)、[taichi/runtime/amdgpu/jit_amdgpu.cpp](taichi/runtime/amdgpu/jit_amdgpu.cpp)。
- **机制**：`fast` tier 把 LLVM opt level 降到 O0；但 NVPTX / AMDGCN 在 O0 下会触发 `Cannot select: i64,ch = stacksave` 致命错误（`StackSave/StackRestore` intrinsic 需要 O1 级 legalization），因此通过 `min_level=1` 强制下限。
- **编译收益**：CPU **+11.4%**（0.615s → 0.545s，bit-exact）、CUDA **+12.1%**（0.909s → 0.800s，bit-exact）。
- **正确性**：简单算术 Δ=0；累加密集型 ≤ 2e-7（仍在 1e-5 跨后端线内）。
- **Offline cache**：`compile_tier` 已纳入 key（P2.c 遗产），fast / balanced 二进制互不冲突。
- **判决依据**：opt-in；收益方向明确；默认 balanced 行为不变。

## 三、回退项

### P3.c · `irpass::scalarize` 早退（**已回退**）

- **原提交**：`274268544`（实现）+ `bfd6871fc`（正确性修复）。
- **机制**：在 `scalarize()` 入口用 `HasMatrixStmt` visitor 预扫 IR，若无矩阵/张量型语句则跳过全部 4 个子 pass（Scalarize / ScalarizePointers / ExtractLocalPointers / FuseMatrixPtr）。
- **回退理由**：
  1. **已验证正确性事故**：原提交 `HasMatrixStmt` 未设 `invoke_default_visitor=true`，导致带类型的矩阵语句全部未被访问，早退在真正需要 scalarize 的内核上误发，LLVM 侧直接 `fatal: fadd on [3 x float]`；虽已由 `bfd6871fc` 修复，但**事故本身证明该设计对未来新增 matrix-family 语句类型存在隐式契约**——任何人添加新的 matrix stmt 而忘了更新这个 visitor，同类 miscompile 会再次静默出现；
  2. **长期维护负担**：`HasMatrixStmt` 相当于对 IR 语句集合的"白名单快照"，与 IR 演进脱耦；
  3. **编译收益在噪声内**：A/B at `unrolling_hard_limit=0`，unroll=128 A=0.1697s / B=0.1776s（**+4.7% 回退**）、unroll=800 A=1.6885s / B=1.6722s（−1%），均在 ±5% bench 噪声内，无方向性信号。
- **回退范围**：
  - `taichi/transforms/scalarize.cpp` 中的 `HasMatrixStmt` 类及其所有调用点；
  - `tests/p3/parity_p3c_matrix.py`、`tests/p3/bench_p3c_scalar.py`、`tests/p3/verify_p3c_earlyexit.py`（针对 P3.c 的专用测试 / bench）；
  - `tests/p3/RESULTS.md` 中的 P3.c 章节（替换为"已回退"说明）。

## 四、"未做"项的风险/收益速览

| 项目 | 编译收益预估 | 风险 | 建议 |
|---|---|---|---|

## 五、流程规范（本次审查产出的约束）

1. **任何改动 IR 语句访问者（BasicStmtVisitor 子类）作为优化前提的 PR**，必须在 PR 描述中声明"所依赖的 stmt 类型集合"，并在 CI 中加入覆盖所有当前子类的回归测试；
2. **任何改动 LLVM opt level / PassManager 构造的 PR**，必须同时跑 CPU + CUDA 两个后端的 sanity 测试；新增的"opt level 捷径"必须通过显式 `min_level` 参数而非隐式默认；
3. **任何 "compile 优化" 类 PR**，若 bench 收益在 ±5% 噪声范围内，默认不予合并，除非其作为其它已合并项的前置依赖。
