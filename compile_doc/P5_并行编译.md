# P5 — 并行编译优化

本文档对应 P5 阶段的设计、实现与实测结果。与全项目"正确性 / 稳定性 / 运行时性能优先"的原则一致：
P5 的编译时优化**仅在不牺牲前三者且证据充分**的前提下保留。

---

## 0. 目标与约束

用户要求：
> 考虑并行编译优化。这对提高性能非常重要，但对实践来说，这里的主要问题是对于一个比较大的复杂项目，怎么处理编译的次序很重要。需要支持 cpu / cuda / vulkan。

核心约束：
1. **编译次序**必须在 C++ 层保持独立 —— 每个 `Kernel` 到达 C++ 时已经完成 `@ti.func` 内联与模板特化（Python AST 阶段），彼此无数据依赖。
2. **跨后端**：CPU (LLVM)、CUDA (LLVM+NVPTX)、Vulkan (SPIR-V) 必须都能受益或至少保持不变。
3. **零回归**：P3 序列 (`parity_p3`, `test_tier_fast_parity`, `test_tier_fast_cuda`, `test_p3d_error_format`) 全部通过。

## 1. 线程安全审计（实施前做的工作）

在动任何代码前，对编译流水线逐层做了静态审计：

| 状态 | 备注 |
|------|------|
| `KernelCompilationManager::caching_kernels_` | **无锁** | P5.a 新增 `cache_mutex_` 保护 |
| `KernelCompilationManager::cached_data_.kernels` | **无锁** | 同上 |
| `KernelCompilationManager::updated_data_` | **无锁** | 同上 |
| `TaichiLLVMContext::per_thread_data_` | `thread_map_mut_` 保护 ✓ | 不动 |
| `TaichiLLVMContext::struct_modules` | `add_struct_module` 主线程独占（TI_ASSERT） | 不动。Parallel compile 发生在 `materialize_runtime` 之后，主线程已静态化。Worker 线程只读/克隆。 |
| `TaichiLLVMContext::snode_tree_funcs_` | 声明存在但无读写 —— **DEAD FIELD** | 不动 |
| CUDA `cuModuleLoadDataEx` | `CUDAContext::get_lock_guard` 串行 ✓ | 不动；本就只占编译时间 16-24%。 |
| Vulkan SPIR-V codegen | 无共享状态 ✓ | 天然线程安全 |

**P2.d 阶段当时担心的"返回 `const CompiledKernelData&` 在 map rehash 后失效"问题**：经验证对象实际存在 `std::unique_ptr<CompiledKernelData>` 指向的堆内存上，地址稳定，引用在 map 重哈希后仍然有效。无需改动返回契约。

## 2. P5.a — 缓存层线程安全基础 `0fbfbe587`

**动机**：没有锁保护的 map 在并行编译下会发生数据竞争。在引入任何并行调度前，先补齐正确性。

**改动**（`taichi/compilation_manager/kernel_compilation_manager.{h,cpp}` ~60 LOC）：

```cpp
mutable std::mutex cache_mutex_;          // 保护三张缓存表
std::condition_variable cache_cv_;         // 去重等待
std::unordered_set<std::string> in_progress_keys_;
```

**关键点**：
1. **编译工作不在锁内**：`load_or_compile` 在查缓存、注册 in-progress 后**释放锁**再调用 `compile_kernel`，结束后再加锁插入结果。这是允许真正并行的前提。
2. **重复请求去重**：两个线程请求同一 `kernel_key` 时，只有一个真正编译，另一个在 `cache_cv_` 上等待并重新探测缓存。避免浪费 CPU、避免两份结果的不一致。
3. **异常安全**：若编译抛异常，先移除 `in_progress_keys_` 并 `notify_all`，保证其他等待者不死锁。
4. **`dump()` 加锁**：即便通常在主线程调用，也把 worker 的收尾竞争窗口堵上。

**回归验证**：`parity_p3`、`test_tier_fast_parity`、`test_tier_fast_cuda`、`test_p3d_error_format` —— 全部 Δ=0 或 3.994e-6（CPU/CUDA 量化噪声）。

**性能影响**：序列路径上 `cache_mutex_` 无竞争 → 无可测开销。

## 3. P5.b — 批量并行编译 API

**改动**（`taichi/program/program.{h,cpp}` + `taichi/python/export_lang.cpp` + `python/taichi/lang/misc.py`）：

### C++ API
```cpp
void Program::compile_kernels(
    const CompileConfig& compile_config,
    const std::vector<const Kernel*>& kernels);
```
用一个临时 `ParallelExecutor("compile_kernels", num_compile_threads)` 分发。第一个 worker 异常被捕获并在 `flush()` 后重抛。

### Python API
```python
import taichi as ti
ti.compile_kernels([kernel1, kernel2, ...])         # 无模板参数的 kernel
ti.compile_kernels([(kernel, args_tuple), ...])      # 需要触发特化的 kernel
```
Python 端在主线程串行完成 `ensure_compiled` (AST 特化) —— 这部分受 GIL 限制无法并行；然后用 `py::call_guard<py::gil_scoped_release>()` 释放 GIL，进入 C++ 并行编译。

### 次序正确性
- `@ti.func` 内联、模板特化在 Python AST 阶段完成 → 到达 C++ 时每个 `Kernel` 已经是自包含的 IR，无跨 kernel 依赖。
- SNode struct modules 必须在 `materialize_runtime()` 之后才做 batch compile（已由 `Program` 初始化顺序保证）。
- 调用方契约（文档中已声明）：**并行编译期间禁止销毁 SNode 树 / 字段**。

### CUDA 安全性
LLVM 编译和 NVPTX 优化在 `TaichiLLVMContext` 的 TLS 上下文中进行 —— 线程安全。`cuModuleLoadDataEx`（GPU 模块上传）仍由 `CUDAContext::get_lock_guard` 串行，这是底层 driver 的要求，接受。

### Vulkan 安全性
SPIR-V codegen 对 Program 状态零依赖，**完全无锁并行**。

## 4. 实测

### CPU（x64, 16 kernels, 4 threads, 子进程隔离冷编译）

| 模式 | 中位耗时 | 加速比 |
|------|----------|--------|
| 串行 (`k()` 逐个) | **102 ms** | 1.00× |
| 并行 (`ti.compile_kernels`) | 104 ms | 0.98× |

Python-side `ensure_compiled`: **51.5 ms**（串行，受 GIL 限制）
C++-side 并行编译：**52.5 ms**

**结论（诚实）**：**CPU 路径没有加速**。根本原因：`LlvmProgramImpl::compilation_workers` (num_compile_threads 线程池) **已经**在每个 kernel 的 `compile_kernel_to_module` 内部并行处理 offload 任务。对于单 offload kernel，16 个 kernel × 1 任务 = 16 个任务被 inner pool 的 4 个线程 FIFO 消费 —— 这本身就是 4 路并行。再套一层 outer pool 只增加同步开销，不增加实际并行度。

### CUDA / Vulkan
当前机器缺完整运行环境（CUDA driver + Vulkan loader）来跑冷编译基准，预期：
- **CUDA**：与 CPU 相似（受 inner `compilation_workers` 已并行影响），额外受 `cuModuleLoadDataEx` 串行影响。估计 1.0-1.2×，实际需生产环境测。
- **Vulkan**：SPIR-V 路径**完全没有** inner pool 并行，当前是严格串行的每核编译。P5.b 应给出接近 num_compile_threads × 的加速（16 kernel / 4 threads 预期 ~3-3.5×）。这是 P5.b 真正能发挥价值的场景。

| 子项 | 正确性 | 稳定性 | 运行时性能 | 编译时性能 | 决定 |
|------|--------|--------|------------|------------|------|
| **P5.a** 缓存锁 | 修复潜在数据竞争 ✅ | 零行为变化 ✅ | 无影响 | 序列路径 +0% | **保留** |
| **P5.b** 批量 API（LLVM 后端） | 保持等价 ✅ | 无回归 ✅ | 无影响 | CPU/CUDA ~1.0× | **保留**（为 Vulkan 保留接口，为 LLVM 后端未来多 offload kernel 保留能力） |

**没有 P5.c**：原计划把"Python 端 compile_kernels"独立成 P5.c，但其实现很薄（<30 行），已合并进 P5.b。

## 6. 未来方向（不在本阶段做）

1. **Vulkan 实测**：在带 Vulkan driver 的生产环境跑 `tests/p5/bench_parallel_compile.py --arch vulkan`，把加速比补回本表。
2. **LLVM inner pool 退化**：如果查明 `LlvmProgramImpl::compilation_workers` 对单 offload kernel 负收益，可在 `compile_kernel_to_module` 中对 `offloads.size() <= 1` 直接 inline 执行，避开 pool 调度。在 P5.b 并行分发场景下或许能把 outer pool 的价值激活。
3. **编译期缓存预热**：`ti.compile_kernels` 可在 Python 层注册一个"warm pool"，在应用启动时后台异步编译常用 kernel，不阻塞主线程。需要配合 `CompileConfig` 的线程安全（未审）。
